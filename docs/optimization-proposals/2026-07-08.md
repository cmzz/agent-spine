## 建议 1（本次用户点名）：implement phase 耗时未记账

- **观察**：`spec-report.json` 的 `duration.phases_ms.implement = null`，而同一 change 的 `review-r0=86,725ms`、`archive=3,274ms` 均有值。by-phase agg 里 implement 有 token（in=98,485 / out=102,065, count=109）却无 duration 维度——事件发得出，唯独缺时长。
- **诊断**：in-session deferred coder 路径（`npc implement record`，`src/npc/pipeline.py` implement record 出口）装订时未 stamp phase start/end → `state.progress[*].phases.implement.duration_ms` 恒空。review/archive 走 `npc phase` 计时故有值，implement 漏接。
- **建议**：在 implement record 成功出口补记 phase duration（对齐 review/archive 的 `npc phase` 计时口径）；deferred 场景以 spawn→record 的墙钟或 record 时刻回填。**只描述，不动手。**
- **验证方式**：新 run 的 `spec-report.json` implement duration 非 null；`npc telemetry agg` by-phase 出现 implement 的 p50_duration_ms。

## 建议 2：blocking_categories 未进聚合 → 收敛盲区

- **观察**：所有 phase 的 `top_blocking_categories` 均为 `[]`（by-phase agg 全空）。但主导成本正是 review-fix 长尾，我们却看不见**哪类 finding 反复触发再 review**。
- **诊断**：review.round 事件（或其聚合）未把 blocking category 落到派生指标；`spec.report` 刚接线了 change 级 category，但跨 run agg 维度仍缺。没有这个维度，无法定位"打地鼠"根因（是同类反复漏扫，还是 fix 引入新回归）。
- **建议**：把每轮 review 的 blocking_categories 汇入 by-phase / by-change agg（top-N 分布）。有了它才能针对性改 focus 模板或 coder 根因扫描规则。**只描述，不动手。**
- **验证方式**：`npc telemetry agg` by-change 出现非空 category 分布；据此改 focus 模板后，观察 review-r2+ 的 count 下降（长尾变短）。

## 建议 3（减法/收紧）：max-rounds cap 设过高，形同虚设

- **观察**：52 runs 内 `stale` 触发 8 次、`max-rounds` **0 次**；而 review 长尾实际能走到 r11。默认上限 20（large 亦 20）从未被触及——非收敛的 change 总是先被 `stale` 拦下。
- **诊断**：max-rounds 作为 review-fix 循环的硬顶，因设得远高于真实收敛/停滞点（>99% 在 r11 前收敛或 stale）而成为死配置，既不保护也不省 token。而每多一轮 review+fix ≈ 数十 k token。
- **建议**：把 `max_rounds` 默认从 20 下调到贴近真实长尾的水平（如 8），让它从死配置变成**真正兜住病态 change token 出血的 backstop**；large change 可保留更高上限。**只描述，不动手。**
- **验证方式**：下调后 `npc telemetry cages` 中 `max-rounds` 从 untriggered 移出（偶发触发即为生效 backstop）；review-r9+ 的 count 归零，长尾 token 支出下降。
