## 建议 1：让 coder 与 record 阶段执行同一条测试命令，消除“交付后才失败”

- **观察**：implement 共 244 次，失败 40 次，失败率 **16.39%**；其中 `rerun-tests-failed=28`，占 implement 失败的 **70%**。implement p50=1,168,469ms（约 19.5 分钟），p95=4,528,386ms（约 75.5 分钟），说明这类失败发生得晚且返工昂贵。
- **诊断**：`src/npc/templates.py::render_implementer` 只要求 coder 参考 `CLAUDE.md / project.md` 自选项目测试命令；真正装订时，`src/npc/pipeline.py::record_implement` 调用 `src/npc/verify.py::run_tests_result`，通过 `resolve_test_cmd` 另行解析并复跑。两端测试命令或运行条件可能不一致，直到 commit 和 summary 已产出后才暴露。
- **建议**：在 implement/fix prompt 渲染时，把 `resolve_test_cmd` 的最终命令作为 runtime truth 注入，并要求 coder 在 commit 前执行该命令、把命令与结果写入 summary；保留 record 阶段的独立复跑硬轨，不降低验证强度。落点：`src/npc/agent.py` 的 prompt 组装、`src/npc/templates.py::render_implementer`/fix prompt、`plugins/agent-spine/agents/spine-coder.md`。
- **验证方式**：连续 30d 中 implement failure rate 从 16.39% 降到 <8%；`rerun-tests-failed` 从 28 降到 <10；同时 record 阶段复跑仍保持开启且不存在“靠跳过测试换成功”的回归。

## 建议 2：在 spec write/fix 结束前增加跨 artifact 一致性清点

- **观察**：43 个 `spec_review-r0` 全部 `changes-requested`，blocking=124，其中 `contradiction=31`、`missing-scenario=21`。进入 fix 后矛盾没有快速消失：`contradiction` 在 r1/r2/r3/r4 分别仍为 **30/22/18/18**；r4 仍有 24 个 change、52 条 blocking。按周看，W29 的首要类别已变成 `contradiction=105`，其次是 `missing-scenario=70`。
- **诊断**：`src/npc/templates.py::render_spec_writer` 强调模式映射和多落点枚举，`render_spec_fixer` 主要嵌入上一轮 finding；但没有要求 writer 在交付前建立“同一行为决策在 proposal/design/tasks/spec 中的唯一口径”并逐项核对，也没有统一的 Requirement→正常/错误/中止/部分失败 Scenario 清点。逐 finding 修补容易在另一份 artifact 留下旧说法，或修一处后产生新的跨文件矛盾。
- **建议**：给 writer/fixer 增加一个**通用的一致性收尾步骤**：从现有 artifact 自主提取行为决策与 Requirement，逐项核对 proposal→design→tasks→spec 的表述和 Scenario 覆盖，再允许输出 RESULT；fix 轮还应对已签发 finding 的同类落点做全 artifact 扫描。只注入通用自检方法和上一轮已签发 findings，**不把 reviewer rubric/category 枚举泄漏给生成侧**。落点：`src/npc/templates.py::render_spec_writer`、`render_spec_fixer` 与 `plugins/agent-spine/agents/spine-spec-writer.md`。
- **验证方式**：同等样本下 `spec_review-r0` 的 contradiction 从 31/43 降到 <15/43，r1 contradiction 降到 <10，进入 `spec_review-r3+` 的 change 数减少至少 50%；`openspec validate --strict` 通过率不得下降。

## 建议 3：把 cage 的“触发机会”和“auto-decide 已执行”分开统计，再决定删除

- **观察**：169 个 run 下，`cages` 把 6 个零触发项列为 deletion candidates；但聚合指标同时出现了对应故障：`codex-exec-failed` 至少出现在 code/spec review，`summary-missing` 在 phase 聚合中出现 2 次，`commit-not-found` 出现 2 次，archive phase 失败 15 次。也就是说，“笼子计数为 0”不等于“没有需要该笼子的故障机会”。
- **诊断**：`src/npc/telemetry.py::_CAGE_DEFS` 对这 6 个笼子只统计 `auto_decide.decision` 的 trigger；`cage_stats`/`cli_cages` 又把 decision count=0 直接列入 `deletion_candidates`。当前口径混淆了三种状态：上游从未发生故障、发生故障但未路由到 auto-decide、已路由并完成裁决。后两者被压成 0 时，删除建议可能恰好删掉一条未接通或被绕过的安全路径。
- **建议**：为每个 cage 同时定义 `opportunity` 来源和 `decision` 来源：前者来自 `phase.exit` / `review.round` / `spec_review.round` 的对应失败原因，后者继续来自 `auto_decide.decision`。输出 `opportunities`、`decisions`、`routing_gaps`；只有 opportunity=0 且 decision=0、样本量达阈值时才进入 deletion candidates，opportunity>0 且 decision=0 则列为接线/编排缺口。落点：`src/npc/telemetry.py::_CAGE_DEFS`、`cage_stats`、`cli_cages`。
- **验证方式**：用现有 90d 派生数据重算后，存在对应失败的 cage 不再被列为删除候选，而进入 `routing_gaps`；新增单测覆盖三态；修复编排后相同故障应满足 decision/opportunity 比例接近 1。
