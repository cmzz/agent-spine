# Spine 自迭代分析 — 2026-07-20

**触发问题**：为什么执行记录通常要 fix 7 到十几轮？

**数据来源**（只读派生指标）：
- `npc telemetry hotspots --top 5 --since 30d`（events_considered=2273）
- `npc telemetry agg --since 30d`
- `npc telemetry cages --since 90d`（runs_observed=183）
- `npc telemetry tail --kind review.round --last 20`（派生 record 采样，非原文）

---

## 核心发现：不是"修不好"，是 review 每轮都在产生新发现，收敛靠 reviewer "扫无可扫"

各轮 changes-requested 率几乎是一条水平线，不随轮次下降：

| 轮次 | review 数 | changes-requested | CR 率 | blocking 总数 | 平均 blocking/review |
|---|---|---|---|---|---|
| r0 | 202 | 176 | 87% | 498 | 2.46 |
| r1 | 164 | 148 | 90% | 227 | 1.38 |
| r2 | 111 | 102 | 92% | 141 | 1.27 |
| r3 | 66 | 61 | 92% | 70 | 1.06 |
| r4 | 50 | 47 | 94% | 50 | 1.00 |
| r5 | 29 | 26 | 90% | 33 | 1.14 |
| r6 | 16 | 15 | 94% | 22 | 1.37 |
| r7–r11 | 16 | 16 | 100% | 15 | ~1 |

平均每轮 blocking 从 2.46 收敛到 ~1 后**再也不降到 0**：reviewer 几乎每轮都能找出"至少 1 条 blocking"，而 1 条 blocking = changes-requested = 再来一轮。tail 采样里的 `composer-single-input-routing` 是典型标本：r0=2 条 blocking → r4 反而 **5 条**（新冒出 billing 类）→ r5/r6 又各 2 条（新冒出 security 类）。晚轮发现的不是"上一轮没修好"，而是**上一轮根本没提过的新类别存量问题**——每轮都是全量新扫，不是 delta 验证。

r≥3 的 review 共 177 次，占全部 654 次 review 事件的 27%，对应约 100+ 次 fix 轮（fix p50 每轮 7–9 分钟、review p50 ~4 分钟）——这就是"7 到十几轮"的直接来源。

---

## 建议 1（最高杠杆）：晚轮切 delta-review 模式 + 显式收敛规则

- **观察**：CR 率 r0→r9 恒在 87–100%；blocking/review 从 r3 起平台在 ~1.0–1.4，永不归零；采样标本 r4 blocking 数（5）多于 r0（2）且是全新类别。
- **诊断**：review focus 模板（`src/npc/focus.py` 生成的 `round-N.focus.md`）对每一轮 reviewer 的指令都是同构的全量审查，没有"本轮只验证上轮 blocking 是否修复 + diff 新引入问题"的约束；且 verdict 规则是"任意 1 条 blocking 即 changes-requested"，无收敛判据。adversarial pass 只在 r0 跑（采样中 r≥1 全部 `adversarial_pass_ran=false`），说明架构上已经有"轮次分层"的先例，但主 review 没有分层。
- **建议**：r≥2 的 focus 模板切换为 delta 模式——(a) 逐条核验上轮 blocking 的修复状态；(b) 只对本轮 fix diff 引入的新问题可判 blocking；(c) 存量代码里新发现的问题一律降级 advisory（随 archive 记录，不阻塞）。可再加一条硬收敛规则：连续两轮无"上轮遗留未修复"即 approve（advisory 照常携带）。**只描述，不动手改。**
- **验证方式**：`npc telemetry agg` 中 review_rounds p50 下降；r≥3 review 事件占比从 27% 降到 <10%；每 change 的 fix 轮数 p50 从 7+ 降到 ≤3。

## 建议 2：validation 类 blocking 一家独大——coder 端前置消化 + reviewer 端提高 blocking 门槛

- **观察**：validation 是每一轮的 top blocking 类别且断层领先：r0 146/498（29%）、r1 120/227（53%）、r2 82/141（58%）、r3 44/70（63%）——轮次越晚占比越高，说明它是晚轮"总能再找出一条"的主要弹药库。另外 r0 的 spec_attribution 里 spec-silent=78（16%），reviewer 在 spec 没写的地方用自己的标准补位判罚。
- **诊断**：两头都有问题。coder 侧：`src/npc/coder.py` / `src/npc/templates.py` 渲染的实现 prompt 没有系统性的输入校验 checklist，r0 就欠下 146 条债。reviewer 侧：validation 类 finding 无严重度门槛，"理论上可能传入 None"级别的 nit 也判 blocking，且 spec 未覆盖的校验要求（spec-silent）同样判 blocking。
- **建议**：(a) coder prompt 模板加一段固定的 validation 自查清单（边界值 / None / 类型 / 外部输入），在 r0 前置消化；(b) reviewer focus 模板要求 validation 类 blocking 必须附"可触发的具体输入/调用路径"，否则降级 advisory；(c) spec-silent 归因的 finding 默认 advisory，让 spec 补条款而不是 fix 循环买单。**只描述，不动手改。**
- **验证方式**：r0 blocking_total（现 498）与 validation 占比同降；spec-silent 归因的 blocking 数从 78 趋近 0（转为 advisory 或 spec 变更）。

## 建议 3（减法）：6 个笼子 90d / 183 runs 零触发，候选删除

- **观察**：`npc telemetry cages --since 90d` 中 `max-rounds`、`agent-timeout-exhausted`、`codex-failed`、`summary-missing`、`commit-not-found`、`archive-failed` 六个硬轨在 183 个 run（远超 min_runs_threshold=5）内触发 0 次，已被列为 deletion_candidates。其中 `max-rounds` 零触发与"实际跑到 r11"并存，说明上限设得太高，从未起到止损作用——建议 1 落地后它更应该改成**低阈值真闸门**（如 5 轮）而非删除。
- **诊断**：这些守卫逻辑仍在 pipeline 代码中但从未被业务路径触及，属冗余分支或场景已被上游防御（如 review round 1 的 commit 核验链）覆盖。注意 `routing-violation` / `timeout-budget` / `record-timeout` 是 no_data（未接线），不可误判为可删。
- **建议**：除 `max-rounds`（改为下调阈值保留）外，其余 5 个逐一评估删除或软化触发条件（定位在 `src/npc/pipeline.py` 各 cage 分支）。**只描述，不动手删。**
- **验证方式**：删/改后 `npc telemetry cages` 中对应项移出 deletion_candidates；全量 pytest 无回归；下调 max-rounds 后该笼子开始出现少量真实触发（说明闸门生效）。
