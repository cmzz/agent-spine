## MODIFIED Requirements

### Requirement: Kimi generation follows generic backend-aware review routing; kimi remains an invalid review engine value

The system SHALL route review of Kimi-generated code and Kimi-written specifications using the same generic, backend-aware review-engine routing that applies to every supported coder backend: unless a review-engine override is supplied, review defaults to `codex` for Kimi-generated artifacts — the same default that applies to Claude- and MiMo-generated artifacts (only Codex-generated artifacts default to Claude). An explicitly supplied review engine that is a member of the supported engine set (`claude`, `codex`) MUST be honored and executed for a Kimi-generated artifact; the system MUST NOT reject it solely because the actual generator was Kimi. Explicit selection remains subject only to the existing shared structural invariants — generation⊥verification orthogonality (self-review) and the MiMo exec-only restriction. The Kimi generator identity itself never trips either invariant: self-review requires the reviewer to equal the generator, and `kimi` is never a valid review engine, so a Kimi-generated or Kimi-written artifact can never be self-reviewed; and Kimi generation identity carries no `mimo` signal, so it never triggers MiMo exec-only on its own. This exemption is about generator identity only — it does not exempt the artifact from MiMo exec-only when the *review configuration itself* is what matches: if the explicitly selected review engine, or an explicitly configured `claude_model`/`claude_bin`, contains the substring `mimo`, the review is still rejected with `mimo_exec_only`/`spec_mimo_exec_only` per the generic shared rule, exactly as it would be for any other generator backend. Independent of generator identity, `kimi` itself remains categorically invalid as a `review.engine` or `spec_review.engine` value and MUST always be rejected as an unknown review engine before any generator-identity check is evaluated.

#### Scenario: Explicit kimi review engine is rejected as unsupported, independent of the generator's identity
- **WHEN** a review is explicitly requested with review engine `kimi`, for code or for a specification, regardless of which backend generated the artifact under review
- **THEN** the review does not execute
- **AND** the result reports an unknown review engine error

#### Scenario: Default code review after Kimi coding routes to Codex
- **WHEN** the latest implementation or fix artifact was generated by Kimi and no review engine override is supplied
- **THEN** the review engine is Codex

#### Scenario: Explicit Claude code review is honored for Kimi-generated artifacts
- **WHEN** the latest implementation or fix artifact was generated by Kimi and code review is explicitly requested with the supported engine `claude`
- **THEN** the review executes using the Claude engine
- **AND** the result is not reported as a routing violation

#### Scenario: Explicit Codex code review is honored for Kimi-generated artifacts
- **WHEN** the latest implementation or fix artifact was generated by Kimi and code review is explicitly requested with the supported engine `codex`
- **THEN** the review executes using the Codex engine (matching the unconfigured default, but now also reachable via explicit request)
- **AND** the result is not reported as a routing violation

#### Scenario: Default spec review after Kimi writing routes to Codex
- **WHEN** a specification was written in a Kimi runtime and no spec review engine override is supplied
- **THEN** the spec review engine is Codex

#### Scenario: Explicit Claude spec review is honored for a Kimi-written specification
- **WHEN** a specification was written in a Kimi runtime and spec review is explicitly requested with the supported engine `claude`
- **THEN** the spec review executes using the Claude engine
- **AND** the result is not reported as a spec routing violation

#### Scenario: Non-Kimi generation keeps existing review routing
- **WHEN** the actual generator backend is not Kimi (including Claude, Codex, or MiMo)
- **THEN** review engine resolution follows the existing generic backend-aware default and explicit-override precedence, unaffected by Kimi-specific handling

#### Scenario: Unavailable or failing Claude reviewer does not fall back to self-review when explicitly selected for Kimi-generated code
- **WHEN** the latest implementation or fix artifact was generated by Kimi, review is explicitly requested with the Claude engine, and the Claude CLI binary cannot be located or the Claude review process exits with a failure
- **THEN** the review does not execute
- **AND** the result reports the existing structured dependency-missing error or Claude execution-failure error, matching which condition occurred
- **AND** no Codex or Kimi review is executed as a substitute, and no result is reported as a passing review

#### Scenario: Unavailable or failing Claude reviewer does not fall back to self-review when explicitly selected for a Kimi-written specification
- **WHEN** a specification was written in a Kimi runtime, spec review is explicitly requested with the Claude engine, and the Claude CLI is either unavailable or the Claude spec review process fails
- **THEN** the spec review does not execute
- **AND** the result reports the existing structured dependency-missing error or Claude execution-failure error, matching which condition occurred
- **AND** no Codex or Kimi spec review is executed as a substitute, and no result is reported as a passing review
