## ADDED Requirements

### Requirement: Codex-native plugin entrypoints
The plugin SHALL be natively discoverable by Codex and SHALL expose run, spec, and analyze skills that execute the same canonical workflows as the existing Claude entrypoints.

#### Scenario: Codex discovers all workflows
- **WHEN** a user installs the agent-spine plugin in Codex
- **THEN** Codex discovers native skills for run, spec, and analyze
- **AND** each skill identifies the corresponding canonical workflow as its source of behavior

#### Scenario: Claude entrypoints remain unchanged
- **WHEN** the same plugin is loaded by Claude Code
- **THEN** the existing run, spec, analyze commands and agent definitions remain available with their existing semantics

### Requirement: Runtime host identity is persistent and backward compatible
The system SHALL persist the explicitly selected runtime host for a run and MUST interpret omitted or legacy runtime-host metadata as Claude.

#### Scenario: Codex initializes a run
- **WHEN** Codex initializes a run with runtime host `codex`
- **THEN** the run metadata and initialization result identify the runtime host as `codex`

#### Scenario: Existing initialization remains Claude
- **WHEN** initialization omits a runtime host or an existing run lacks that field
- **THEN** the effective runtime host is `claude`

### Requirement: Codex native generation uses in-session agents
The system SHALL use Codex as the unconfigured default coder and spec writer inside a Codex runtime, and SHALL dispatch the default Codex coder in-session without invoking a headless Codex coder process.

#### Scenario: Unconfigured Codex code generation
- **WHEN** an implement or fix phase runs in a Codex runtime without an explicit coder backend or dispatch
- **THEN** the selected backend is `codex`
- **AND** the phase returns a deferred in-session agent request

#### Scenario: Explicit routing remains authoritative
- **WHEN** a coder backend or dispatch is explicitly supplied by CLI or project configuration
- **THEN** the existing CLI and configuration precedence remains authoritative

#### Scenario: Unsatisfiable explicit spec writer is rejected
- **WHEN** a Codex runtime has an explicit spec writer backend configured to a non-Codex value
- **THEN** spec pipeline entrypoints report a spec routing violation identifying the host mismatch
- **AND** the configured value is not silently ignored or rewritten

#### Scenario: Claude default is unchanged
- **WHEN** an implement or fix phase runs in a Claude runtime without explicit routing
- **THEN** backend and dispatch resolve exactly as they did before Codex runtime support

### Requirement: Codex generation is reviewed by Claude
The system MUST route LLM review of Codex-generated code and specifications to Claude and MUST reject an explicit Codex review of a Codex-generated artifact.

#### Scenario: Default code review after Codex coding
- **WHEN** the latest implementation or fix artifact was generated by Codex and no review engine override is supplied
- **THEN** the review engine is Claude

#### Scenario: Explicit same-source code review is rejected
- **WHEN** the latest implementation or fix artifact was generated by Codex and review is explicitly requested with Codex
- **THEN** review does not execute
- **AND** the result reports a routing violation

#### Scenario: Default spec review after Codex writing
- **WHEN** a specification was written in a Codex runtime and no spec review engine override is supplied
- **THEN** the spec review engine is Claude

#### Scenario: Explicit same-source spec review is rejected
- **WHEN** a specification was written in a Codex runtime and spec review is explicitly requested with Codex
- **THEN** review does not execute
- **AND** the result reports a spec routing violation

#### Scenario: Non-Codex generation keeps existing review routing
- **WHEN** the actual generator backend is not Codex
- **THEN** review engine resolution follows the existing CLI and configuration precedence

### Requirement: Actual generator identity is retained for routing
The system SHALL retain the actual backend selected for each new implement and fix phase and SHALL use that identity when enforcing generation-review orthogonality.

#### Scenario: New coder phase records backend
- **WHEN** an implement or fix phase starts with a selected backend
- **THEN** that backend remains present in the completed phase record

#### Scenario: Legacy phase remains reviewable
- **WHEN** a historical phase has no recorded generator backend
- **THEN** review falls back to the existing configuration-based backend resolution

### Requirement: Hook and session behavior is host compatible
The plugin SHALL make Codex session metadata available to the existing run initialization path and MUST NOT reject a Codex sub-agent solely because its stop payload omits Claude-specific message fields.

#### Scenario: Codex session start is indexed
- **WHEN** Codex invokes the plugin session-start hook with session, working-directory, and transcript metadata
- **THEN** subsequent initialization can resolve that session through the shared session index

#### Scenario: Codex stop payload omits Claude message field
- **WHEN** a Codex sub-agent stop payload does not contain the Claude-specific final-message field
- **THEN** the compatibility hook exits successfully
- **AND** the deterministic record command remains responsible for result validation

#### Scenario: Claude stop validation remains active
- **WHEN** a Claude spine coder stop payload contains the existing final-message field
- **THEN** the hook applies the existing RESULT and commit validation behavior

### Requirement: Codex permissions are scoped and explicit
The plugin SHALL document the external directories required by agent-spine and MUST NOT automatically modify user-global Codex configuration or credentials.

#### Scenario: User prepares Codex permissions
- **WHEN** a user follows the Codex installation instructions
- **THEN** the instructions identify the worktree and task-log directories that require scoped access
- **AND** no plugin installation step writes user-global credentials or permission configuration
